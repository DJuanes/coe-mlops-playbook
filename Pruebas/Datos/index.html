    Testing de Datos

    Hasta ahora, hemos utilizado pruebas unitarias y de integraci√≥n para probar las funciones que interact√∫an con nuestros datos, pero no hemos probado la validez de los datos en s√≠. Vamos a utilizar la librer√≠a great expectations para probar c√≥mo se espera que sean nuestros datos. Es una librer√≠a que nos permite crear expectativas sobre el aspecto que deber√≠an tener nuestros datos de forma estandarizada. Tambi√©n proporciona m√≥dulos para conectarse sin problemas con fuentes de datos de backend como sistemas de archivos locales, cloud, bases de datos, etc.

    python -m pip install great-expectations==0.15.15

    Y agregaremos esto a nuestro script setup.py:

    # setup.py

    "pytest==7.1.2",
    "pytest-cov==2.10.1",
    "great-expectations==0.15.15"
]

    Primero cargaremos los datos sobre los que queremos aplicar nuestras expectativas.

    Expectativas

    A la hora de crear expectativas sobre el aspecto que deben tener nuestros datos, debemos pensar en todo nuestro conjunto de datos y en todos los features (columnas) que contiene.

    # Presencia de features espec√≠ficos.

    column_list=["id", "created_on", "title", "description", "tag"]
)

# Combinaciones √∫nicas de features (¬°detecta fugas de datos!)

# Valores faltantes
df.expect_column_values_to_not_be_null(column="tag")

# Valores √∫nicos
df.expect_column_values_to_be_unique(column="id")

# Adherencia al tipo
df.expect_column_values_to_be_of_type(column="title", type_="str")

        "natural-language-processing", "mlops", "time-series"]
df.expect_column_values_to_be_in_set(column="tag", value_set=tags)

    Cada una de estas expectativas crear√° una salida con detalles sobre el √©xito o el fracaso, los valores esperados y observados, las expectativas planteadas, etc. √âstas son s√≥lo algunas de las diferentes expectativas que podemos crear. Aqu√≠ hay otras expectativas populares:

    • relaciones de valores de features con otros valores de features ‚Üí expect_column_pair_values_a_to_be_greater_than_b
    • recuento de filas (exacto o en un rango) de muestras ‚Üí expect_table_row_count_to_be_between
    • estad√≠sticas de valores (media, std, mediana, max, min, suma, etc.) ‚Üí expect_column_mean_to_be_between

    Todas estas expectativas se guardar√°n en great_expectations/expectations. Tambi√©n podemos enumerar las suites con:

    Para editar una suite, podemos ejecutar el siguiente comando:

    Checkpoints Crea checkpoints donde se aplica una suite de expectativas a un activo de datos espec√≠fico. Esta es una gran manera de aplicar program√°ticamente los puntos de control en nuestras fuentes de datos existentes y nuevas. Para nuestro ejemplo, ser√≠a:

great_expectations checkpoint new tags

    Cada llamada de creaci√≥n de checkpoint lanzar√° un notebook en el que podemos definir a qu√© suites aplicar este checkpoint. Tenemos que cambiar las l√≠neas para data_asset_name (sobre qu√© activo de datos ejecutar el checkpoint suite) y expectation_suite_name (nombre de la suite a utilizar). Luego podemos ejecutarlo:

    great_expectations checkpoint run projects
great_expectations checkpoint run tags
great_expectations checkpoint run labeled_projects

    Documentaci√≥n

    Cuando creamos expectativas utilizando la aplicaci√≥n CLI, Great Expectations genera autom√°ticamente documentaci√≥n para nuestras pruebas. Tambi√©n almacena informaci√≥n sobre las ejecuciones de validaci√≥n y sus resultados. Podemos lanzar la documentaci√≥n de los datos generados con el siguiente comando: great_expectations docs build Por defecto, Great Expectations almacena nuestras expectativas, resultados y m√©tricas localmente, pero para producci√≥n, querremos configurar almacenes de metadatos remotos.

    Producci√≥n

    La ventaja de utilizar una biblioteca como great expectations, frente a las sentencias assert aisladas, es que podemos:

    • reducir los esfuerzos redundantes para crear pruebas en todas las modalidades de datos
    • crear autom√°ticamente puntos de control de las pruebas para ejecutarlas a medida que crece nuestro conjunto de datos
    • generar autom√°ticamente documentaci√≥n sobre las expectativas e informar sobre las ejecuciones
    • conectar f√°cilmente con fuentes de datos backend como sistemas de archivos locales, cloud, bases de datos, etc.
